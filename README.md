# Hibernate Envers 101

Spring Boot 3.5 / Java 21 project showing a complete usage of **Hibernate Envers**
to automatically historicize entities into companion audit tables.

The goal is educational: every Envers feature is exercised by a concrete example,
and on application startup an automatic *runner* replays the whole scenario, so
all you need to do to learn is read the logs and inspect the database.

## Prerequisites

- JDK 21
- Maven 3.9+ (or the included `mvnw` wrapper)
- Local PostgreSQL listening on `localhost:5432`
  - user: `postgres`
  - password: `postgres`
  - database: `envers_demo`

Database creation:
```bash
createdb -U postgres envers_demo
```

## Quick start

```bash
./mvnw spring-boot:run
```

On startup:

1. Hibernate creates/updates the domain tables and the audit (`_AUD`) tables,
   plus the `REVINFO` table.
2. `DemoRunner` (if enabled) wipes the tables and replays the full scenario,
   printing each entity's history to the console.
3. The application keeps listening on `:8080` with the REST endpoints active.

## Audit infrastructure

### `CustomRevisionEntity` (maps the `REVINFO` table)

```java
@Entity @Table(name = "REVINFO")
@RevisionEntity(CustomRevisionListener.class)
public class CustomRevisionEntity extends DefaultRevisionEntity {
    @Column(name = "username") private String username;
    // getter/setter
}
```

Extends `DefaultRevisionEntity` ⇒ inherits `id` (revision number, generated by
sequence) and `timestamp`. Adds `username` to record who made the change.

### `CustomRevisionListener`

Envers calls `newRevision(...)` whenever the transaction touches at least one
`@Audited` entity. In the listener we read the user from the `ThreadLocal`:

```java
public void newRevision(Object revisionEntity) {
    ((CustomRevisionEntity) revisionEntity).setUsername(AuditUserContext.getUser());
}
```

### `AuditUserContext` + `AuditUserFilter`

- `AuditUserContext` exposes a `ThreadLocal<String>` with the current user
  (defaults to `"system"` if not set).
- `AuditUserFilter` reads the `X-User` HTTP header on every request and
  publishes it into the `ThreadLocal`. Clears the value at the end of the request.

In a real application the filter would be replaced with the Spring Security
principal (`SecurityContextHolder.getContext().getAuthentication().getName()`).
Here the filter is enough to demonstrate the mechanism via `curl`.

In `DemoRunner` (which does not go through HTTP) the username is set manually
before each transaction with `AuditUserContext.setUser(...)`.

## Domain entities

| Entity | Audited | Notes |
|---|---|---|
| `Customer` | yes (`@Audited`) | every field goes into `customer_aud` |
| `Address` | yes (`@Audited`) | child of `Customer` (`@ManyToOne`); the relationship is also tracked |
| `Product` | partial | class-level `@Audited` + `@NotAudited` on `stockQuantity` |
| `Category` | no | example of an ignored entity. The `Product → Category` relation uses `@Audited(targetAuditMode = NOT_AUDITED)` |

### Three auditing scenarios illustrated

1. **Full audit** (`Customer`): every change produces a row in `customer_aud`.
2. **Selective audit** (`Product`): changing only `stockQuantity` does *not*
   produce a new revision because the field is annotated `@NotAudited`.
3. **Relation to a non-audited entity** (`Product.category`): without
   `@Audited(targetAuditMode = NOT_AUDITED)` Envers would throw an error at
   startup. With this annotation the relation is tracked by storing only the
   category id at the time of the revision.

## Schema generated by Envers

With `ddl-auto=update` Hibernate generates the following tables in PostgreSQL
(in addition to the standard entity DDL):

### `revinfo`

| Column | Type | Meaning |
|---|---|---|
| `rev` | `integer` PK | progressive revision number (global sequence) |
| `revtstmp` | `bigint` | Unix timestamp in millis |
| `username` | `varchar(100)` | added by `CustomRevisionEntity` |

One row in `revinfo` for every transaction that modified at least one
`@Audited` entity. Revisions are **global**, not per-entity: the same `rev`
can appear in multiple `_AUD` tables.

### `customer_aud` (same shape for `address_aud`, `product_aud`)

| Column | Type | Meaning |
|---|---|---|
| `rev` | `integer` FK→`revinfo.rev` | revision |
| `revtype` | `smallint` | 0 = ADD, 1 = MOD, 2 = DEL |
| `id` | PK type | entity id |
| *audited fields* | as in the entity | snapshot at the revision |
| *`_MOD` columns* | `boolean` | (only with `global_with_modified_flag=true`) `true` if the field changed in that revision |

Composite primary key: `(rev, id)`.

`product_aud` does **not** contain the `stock_quantity` column, because it is
annotated `@NotAudited`.

## Automatic startup demo

`DemoRunner` runs 13 transactions in sequence. Every transaction that touches
an `@Audited` entity produces a new `rev` in `revinfo`. Transactions that
touch only `Category` do not produce revisions.

Below, for each step, you can see the state of the tables right after the
transaction. New or modified rows are highlighted.

### Starting point (with `demo.reset=true`)

`TRUNCATE TABLE address_aud, customer_aud, product_aud, address, customer, product, category, revinfo RESTART IDENTITY CASCADE`

All tables empty. ID and `rev` sequences restart from 1.

### Step 1 — `[alice]` CREATE Customer

Application call:
```java
new Customer("Mario", "Rossi", "mario.rossi@example.com")
```

SQL:
```sql
INSERT INTO customer (first_name, last_name, email) VALUES ('Mario','Rossi','mario.rossi@example.com');
INSERT INTO revinfo  (rev, revtstmp, username) VALUES (1, ..., 'alice');
INSERT INTO customer_aud (rev, revtype, id, first_name, last_name, email) VALUES (1, 0, 1, 'Mario','Rossi','mario.rossi@example.com');
```

`customer`:

| id | first_name | last_name | email |
|---|---|---|---|
| 1 | Mario | Rossi | mario.rossi@example.com |

`customer_aud`:

| rev | revtype | id | first_name | last_name | email |
|---|---|---|---|---|---|
| 1 | 0 (ADD) | 1 | Mario | Rossi | mario.rossi@example.com |

`revinfo`:

| rev | revtstmp | username |
|---|---|---|
| 1 | t₁ | alice |

### Step 2 — `[alice]` ADD Address

Operation: a new `Address` is added to the `Customer` via cascade.

SQL:
```sql
INSERT INTO address (street, city, zip_code, customer_id) VALUES ('Via Roma 1','Milano','20100',1);
INSERT INTO revinfo (rev, revtstmp, username) VALUES (2, ..., 'alice');
INSERT INTO address_aud (rev, revtype, id, street, city, zip_code, customer_id) VALUES (2, 0, 1, 'Via Roma 1','Milano','20100',1);
```

`address`:

| id | street | city | zip_code | customer_id |
|---|---|---|---|---|
| 1 | Via Roma 1 | Milano | 20100 | 1 |

`address_aud`:

| rev | revtype | id | street | city | zip_code | customer_id |
|---|---|---|---|---|---|---|
| 2 | 0 (ADD) | 1 | Via Roma 1 | Milano | 20100 | 1 |

`customer_aud`: **unchanged** (the relationship is `mappedBy` on the `Address`
side, so no UPDATE is issued on `customer`).

`revinfo`:

| rev | revtstmp | username |
|---|---|---|
| 1 | t₁ | alice |
| **2** | **t₂** | **alice** |

---

### Step 3 — `[bob]` UPDATE lastname Rossi → Bianchi

SQL:
```sql
UPDATE customer SET last_name='Bianchi' WHERE id=1;
INSERT INTO revinfo (rev, ..., username) VALUES (3, ..., 'bob');
INSERT INTO customer_aud (rev, revtype, id, first_name, last_name, email) VALUES (3, 1, 1, 'Mario','Bianchi','mario.rossi@example.com');
```

`customer`:

| id | first_name | last_name | email |
|---|---|---|---|
| 1 | Mario | **Bianchi** | mario.rossi@example.com |

`customer_aud`:

| rev | revtype | id | first_name | last_name | email |
|---|---|---|---|---|---|
| 1 | 0 | 1 | Mario | Rossi | mario.rossi@example.com |
| 2 | (no row) | — | — | — | — |
| **3** | **1 (MOD)** | **1** | **Mario** | **Bianchi** | **mario.rossi@example.com** |

> Note: there is no row at rev 2 in `customer_aud` because `Customer` was not
> modified in that revision (only `Address` changed).

`revinfo` adds `(3, t₃, bob)`.

---

### Step 4 — `[carol]` UPDATE email

Before running, the runner captures `Instant beforeFinalEdit = Instant.now()`,
to be used later for the "snapshot at instant" query.

SQL:
```sql
UPDATE customer SET email='mario.bianchi@example.com' WHERE id=1;
INSERT INTO revinfo (rev, ..., username) VALUES (4, ..., 'carol');
INSERT INTO customer_aud (rev, revtype, id, ...) VALUES (4, 1, 1, 'Mario','Bianchi','mario.bianchi@example.com');
```

`customer`:

| id | first_name | last_name | email |
|---|---|---|---|
| 1 | Mario | Bianchi | **mario.bianchi@example.com** |

`customer_aud` now contains 3 rows for `id=1`: rev 1 (ADD), rev 3 (MOD lastname),
rev 4 (MOD email).

`revinfo` adds `(4, t₄, carol)`.

---

### Step 5 — Read: full history

The runner calls `customerService.getHistory(1)` which internally does:

```java
auditReader.createQuery()
    .forRevisionsOfEntity(Customer.class, false, true)
    .add(AuditEntity.id().eq(1))
    .addOrder(AuditEntity.revisionNumber().asc())
    .getResultList();
```

Console output:

```
rev=1 ADD by alice -> Mario Rossi   <mario.rossi@example.com>
rev=3 MOD by bob   -> Mario Bianchi <mario.rossi@example.com>
rev=4 MOD by carol -> Mario Bianchi <mario.bianchi@example.com>
```

> No row for rev 2: `Customer` did not change. The `Address` history at rev 2
> is available separately in `address_aud`.

### Step 6 — Read: snapshot at rev 1

`customerService.getAtRevision(1, 1)` ⇒ `auditReader.find(Customer.class, 1, 1)`.

SQL produced by Envers:
```sql
SELECT * FROM customer_aud WHERE id=1 AND rev<=1 ORDER BY rev DESC LIMIT 1;
```

Result: `Mario Rossi <mario.rossi@example.com>`.

### Step 7 — Read: snapshot at timestamp `beforeFinalEdit`

`customerService.getAtDate(1, beforeFinalEdit)` ⇒ Envers first looks up the
revision active at the timestamp:

```sql
SELECT max(rev) FROM revinfo WHERE revtstmp <= :ts;
```

It finds `rev = 3` (rev 4 is after the timestamp). Then `find(Customer.class, 1, 3)`
returns `Mario Bianchi <mario.rossi@example.com>`.

### Step 8 — Read: diff between rev 1 and rev 4

Console output:

```
lastName: 'Rossi' -> 'Bianchi'
email:    'mario.rossi@example.com' -> 'mario.bianchi@example.com'
```

`firstName` is not listed because it is identical in the two revisions.

---

### Step 9 — `[admin]` REVERT to rev 1

`customerService.revertTo(1, 1)` retrieves the snapshot at rev 1 and re-applies
it as an UPDATE on the current row. It is an **application-level** rollback:
it produces a new revision without altering the existing ones.

SQL:
```sql
UPDATE customer SET first_name='Mario', last_name='Rossi', email='mario.rossi@example.com' WHERE id=1;
INSERT INTO revinfo (rev, ..., username) VALUES (5, ..., 'admin');
INSERT INTO customer_aud (rev, revtype, id, ...) VALUES (5, 1, 1, 'Mario','Rossi','mario.rossi@example.com');
```

`customer`:

| id | first_name | last_name | email |
|---|---|---|---|
| 1 | Mario | **Rossi** | **mario.rossi@example.com** |

`customer_aud` (4 rows for id=1):

| rev | revtype | first_name | last_name | email |
|---|---|---|---|---|
| 1 | 0 | Mario | Rossi | mario.rossi@example.com |
| 3 | 1 | Mario | Bianchi | mario.rossi@example.com |
| 4 | 1 | Mario | Bianchi | mario.bianchi@example.com |
| **5** | **1** | **Mario** | **Rossi** | **mario.rossi@example.com** |

---

### Step 10 — `[admin]` DELETE Customer

Deletion of `Customer` with cascade onto `Address`.

SQL:
```sql
DELETE FROM address WHERE customer_id=1;
DELETE FROM customer WHERE id=1;
INSERT INTO revinfo (rev, ..., username) VALUES (6, ..., 'admin');
INSERT INTO customer_aud (rev, revtype, id, first_name, last_name, email) VALUES (6, 2, 1, 'Mario','Rossi','mario.rossi@example.com');
INSERT INTO address_aud  (rev, revtype, id, street, city, zip_code, customer_id) VALUES (6, 2, 1, 'Via Roma 1','Milano','20100',1);
```

`customer` and `address`: **empty** (rows with `id=1` no longer exist).

`customer_aud` has 5 rows for `id=1`: the new one is at rev 6 with
`revtype=2` (DEL). Thanks to `store_data_at_delete=true` the DEL row holds the
last known values, not `NULL`.

`address_aud` likewise: 2 rows for `id=1` (rev 2 ADD, rev 6 DEL).

**Insight**: the history stays intact. Any past state can be reconstructed
even after the live entity has been deleted.

### Steps 11..13 — Product scenario

Revisions here start from `rev=7` because the `revinfo` sequence is global.

#### Step 11.a — `[alice]` create Categories Electronics and Books

```sql
INSERT INTO category (name) VALUES ('Electronics');  -- id=1
INSERT INTO category (name) VALUES ('Books');        -- id=2
```

**No row in `revinfo`**: `Category` is not `@Audited`.

#### Step 11.b — `[alice]` create Product `Laptop`

```sql
INSERT INTO product (code, name, price, stock_quantity, category_id) VALUES ('LP-001','Laptop',1000.00,10,1);  -- id=1
INSERT INTO revinfo (rev, ..., username) VALUES (7, ..., 'alice');
INSERT INTO product_aud (rev, revtype, id, code, name, price, category_id) VALUES (7, 0, 1, 'LP-001','Laptop',1000.00,1);
```

> `product_aud` does **not** have a `stock_quantity` column because it is `@NotAudited`.

`product`:

| id | code | name | price | stock_quantity | category_id |
|---|---|---|---|---|---|
| 1 | LP-001 | Laptop | 1000.00 | 10 | 1 |

`product_aud`:

| rev | revtype | id | code | name | price | category_id |
|---|---|---|---|---|---|---|
| 7 | 0 | 1 | LP-001 | Laptop | 1000.00 | 1 |

#### Step 11.c — `[alice]` create Product `Effective Java` → rev 8

`product_aud` adds `(8, 0, 2, 'BK-001', 'Effective Java', 45.00, 2)`.

#### Step 12 — `[bob]` updatePrice 1000 → 1200

```sql
UPDATE product SET price=1200.00 WHERE id=1;
INSERT INTO revinfo (rev, ..., username) VALUES (9, ..., 'bob');
INSERT INTO product_aud (rev, revtype, id, code, name, price, category_id) VALUES (9, 1, 1, 'LP-001','Laptop',1200.00,1);
```

`product` updated. `product_aud` now has 2 rows for `id=1` (rev 7 ADD, rev 9 MOD).

#### Step 13 — `[bob]` updateStock 10 → 5 (NO REVISION)

```sql
UPDATE product SET stock_quantity=5 WHERE id=1;
-- NO insert in revinfo
-- NO insert in product_aud
```

**This is the key point of selective auditing.** Hibernate still issues the
UPDATE on the `product` table, but Envers checks the audited fields and, seeing
none of them changed, **does not produce any new revision**.

`product` reflects the new stock=5 but `product_aud` still has 2 rows.

#### Step 14 — `[carol]` updatePrice 1200 → 1500 → rev 10

`product_aud` adds `(10, 1, 1, 'LP-001', 'Laptop', 1500.00, 1)`.

#### Step 15 — Read: query with `price > 1300` criterion

`productService.findPriceChangesAbove(1, 1300)` runs:

```java
auditReader.createQuery()
    .forRevisionsOfEntity(Product.class, false, true)
    .add(AuditEntity.id().eq(1))
    .add(AuditEntity.revisionType().eq(RevisionType.MOD))
    .add(AuditEntity.property("price").gt(new BigDecimal("1300")))
    .getResultList();
```

Generated SQL (simplified):
```sql
SELECT pa.*, r.*, pa.revtype
FROM product_aud pa JOIN revinfo r ON pa.rev = r.rev
WHERE pa.id = 1 AND pa.revtype = 1 AND pa.price > 1300;
```

Only rev 10 satisfies the criterion.

#### Step 16 — Read: snapshot by category at the latest revision

`productService.snapshotByCategory(10, "Electronics")`:

```java
auditReader.createQuery()
    .forEntitiesAtRevision(Product.class, 10)
    .add(AuditEntity.relatedId("category").eq(1L))
    .getResultList();
```

Returns all products that, at rev 10, were in the Electronics category:
just the Laptop with price 1500.

the change. Without the header `system` is used.

## Manual queries on the audit tables

You can query the `_AUD` tables directly with any SQL client. Examples:

```sql
-- All revisions with their author
SELECT r.rev, to_timestamp(r.revtstmp/1000) AS when_, r.username
FROM revinfo r
ORDER BY r.rev;

-- Full history of a Customer
SELECT ca.rev, ca.revtype, ca.first_name, ca.last_name, ca.email,
       r.username, to_timestamp(r.revtstmp/1000) AS when_
FROM customer_aud ca
JOIN revinfo r ON r.rev = ca.rev
WHERE ca.id = 1
ORDER BY ca.rev;

-- State of a product at a specific revision
SELECT * FROM product_aud
WHERE id = 1 AND rev = (
    SELECT max(rev) FROM product_aud WHERE id = 1 AND rev <= 9
);

-- All DELETE operations performed
SELECT 'customer' AS entity, ca.id, r.username, to_timestamp(r.revtstmp/1000)
FROM customer_aud ca JOIN revinfo r ON r.rev = ca.rev
WHERE ca.revtype = 2
UNION ALL
SELECT 'product', pa.id, r.username, to_timestamp(r.revtstmp/1000)
FROM product_aud pa JOIN revinfo r ON r.rev = pa.rev
WHERE pa.revtype = 2;
```

The `_MOD` columns (present thanks to `global_with_modified_flag=true`) make it
easy to filter the revisions in which a specific field changed:

```sql
-- All revisions in which the email of customer 1 changed
SELECT rev, email FROM customer_aud
WHERE id = 1 AND email_mod = true
ORDER BY rev;
```

## Troubleshooting

### "I see the UPDATE on the table but nothing shows up in `_AUD`"

Expected when the only change involves `@NotAudited` fields (e.g.
`Product.stockQuantity`). Envers does not write rows in `_AUD` nor in
`revinfo` when no audited field changed. Also double-check you didn't
accidentally tweak `RelationTargetAuditMode`.

### "Rev N exists in `revinfo` but not in `customer_aud`"

Revisions are global. A given `rev` only appears in those `_AUD` tables whose
entity was modified in that transaction. In the demo scenario, for example,
rev 2 is in `revinfo` and in `address_aud` but not in `customer_aud` (rev 2 =
creation of an Address; the Customer was not touched).

### "Startup error: relation _AUD doesn't reference an audited entity"

Happens when a `@ManyToOne`/`@OneToOne` of an audited entity points to a
*non*-audited entity without qualifying the relationship. Solutions:

- Annotate the target with `@Audited` as well, **or**
- Annotate the relationship with `@Audited(targetAuditMode = RelationTargetAuditMode.NOT_AUDITED)`
  (see `Product.category`).

### "No version of Customer 1 at revision 1" on the second run

`TRUNCATE ... RESTART IDENTITY CASCADE` only resets the sequences owned by
`IDENTITY` columns (e.g. `customer.id`). The REVINFO id, however, is generated
by an independent Postgres sequence (because `DefaultRevisionEntity#id` uses
`@GeneratedValue` AUTO ⇒ SEQUENCE on Postgres). After the first run the rows
are wiped but the sequence keeps growing, so on the second run the customer
is created at e.g. `rev=11` and `getAtRevision(1, 1)` correctly returns nothing.

`DemoRunner.resetSchema()` therefore also resets every standalone sequence:

```sql
DO $$
DECLARE seq_name text;
BEGIN
    FOR seq_name IN
        SELECT c.relname FROM pg_class c WHERE c.relkind = 'S'
    LOOP
        EXECUTE format('ALTER SEQUENCE %I RESTART WITH 1', seq_name);
    END LOOP;
END $$;
```

If you write your own reset routine, remember to do the same.

### "Username on `DemoRunner` revisions is always `system`"

`DemoRunner` must set the `ThreadLocal` *before* opening the transaction
(`AuditUserContext.setUser(...)` *before* `tx.execute(...)`). In the provided
code this is handled by the `asUser(...)` helper. If you add new steps, reuse
that helper.
